1 Apigee Overview

API-First and OpenAPI Specifications

This document outlines key concepts related to REST APIs, API-first development, and OpenAPI specifications, based on the provided material.

REST API Design (Basics)

REST (Representational State Transfer) is an architectural style for designing networked applications. It is not a standard, but a set of guidelines.

Use of HTTP Verbs for CRUD Operations

HTTP verbs are used to specify the type of operation performed on a resource. The four basic functions of persistent storage are Create, Read, Update, and Delete, commonly referred to as CRUD operations.

  • POST (Create):
    • Used to create a new resource.
    • For a collection (e.g., '/dogs'), 'POST/dogs' creates a new dog with an automatically generated ID.
    • It is not a safe method (modifies state) and not idempotent (repeating the call creates multiple resources).
  • GET (Read):
    • Used to retrieve information about a collection of resources (e.g., '/dogs') or a single resource (e.g., '/dogs/1234').
    • It should not update the state of the resource (safe method).
    • It is idempotent (repeating the call produces the same result).
  • PUT/PATCH (Update):
    • PUT:
      • Used to update an existing resource (e.g., `PUT/dogs/1234`).
      • Can also be used to bulk update a collection (e.g., `PUT/dogs`), though this is often not recommended due to risk of accidental mass updates.
      • Not a safe method (modifies state).
      • Idempotent (a second identical 'PUT' call leaves the resource in the same state).
      • Traditionally, 'PUT' completely replaces a resource with the provided payload. However, many public REST APIs use 'PUT' with 'PATCH' semantics (partial updates).
    • PATCH:
      • Used to modify only a subset of a resource (partial update).
      • Fields with a value are added or modified; fields specified as 'null' are deleted; unmentioned fields are unchanged.
      • Consider 'PATCH' for new APIs when partial updates are needed.
  • DELETE (Delete):
    • Used to delete a resource (e.g., `DELETE /dogs/1234`).
    • Can also be used to delete an entire collection or matched resources (e.g., `DELETE /dogs`), though this is often not allowed due to high risk.
    • Not a safe method (modifies state).
    • Idempotent (a second identical 'DELETE' has no effect as the resource is already gone).
Keeping Verbs out of URLs

In REST APIs, URLs should represent noun-oriented resources, not operations.

  • Anti-pattern: `POST/UpdateStudent?id=15`, `/getDog`
  • Recommended: Use primary HTTP verbs (GET, POST, PUT/PATCH, DELETE) with noun-oriented resources (e.g., '/students/15').
  • For non-CRUD operations, consider using `POST` with a query parameter to specify the action (e.g., `POST/dogs/1234?action=walk`).
Sweep Complexity Behind the '?' (Query Parameters)

Use query parameters for filtering, searching, and specifying partial responses to improve developer experience and API performance.

  • Example: Instead of multiple calls to find specific dogs (e.g., 'GET /parks' then 'GET /parks/{parkId}/dogs' then 'GET /dogs/{dogId}' and manual filtering), use `GET /dogs?location=park&color=brown&state=running`.
  • Partial Responses: Allow developers to select specific fields to return in a response using query parameters (e.g., `GET /employees/5678?fields=id,fullName,geo.latitude`). This is beneficial for low-bandwidth apps or large payloads.

API-First Development

API-first development is a strategy where APIs are designed by prioritizing the needs of the app developers who will consume the API.

Core Principles
  • Design from the perspective of API users: Focus on usability and customer use cases, rather than simply exposing existing backend implementations.
  • Design API interface first: The API contract should be defined before implementation begins.
Benefits of API-First Development
  • Shake out issues early: Uncover business and technical gaps with customers and stakeholders before development, avoiding wasted effort on unneeded features or poorly designed backend systems.
  • Promote consistency, usability, and best practices: Early review cycles ensure APIs are consistent, easy to use, and follow best practices, increasing adoption.
  • Increase ability to do parallel development: App developers can design and validate use cases using the API contract even while the API implementation is in progress.

OpenAPI Specifications

What an OpenAPI Specification Does
  • Describes API interface (contract): Defines how apps can use the API.
  • Easily human- and machine-readable: Written in JSON or YAML format.
  • Defines available paths and allowed operations: Specifies endpoints (e.g., `/employees`, `/employees/{id}`) and the HTTP methods allowed for each.
  • Specifies request and response format for each operation:
    • Includes success and failure response codes and payload formats.
    • Describes authentication details.

OpenAPI specifications are leveraged in Apigee for several purposes:

  • Interactive documentation in developer portals: Provides app developers with live documentation and the ability to try out API calls, streamlining the learning process.
  • Generate API proxy stubs: Automatically creates Apigee API proxy stubs with flows for all documented resources and operations.
  • Validate incoming requests: An OpenAPI spec validation policy can be used in an API proxy to ensure incoming requests conform to the specification, rejecting those that don't.

OpenAPI Specifications

OpenAPI specifications are a widely used method to document RESTful APIs.

History

  • WADL (Web Application Description Language): Early XML-based specification for JSON APIs, complex and not human-readable.
  • Swagger (2010): Introduced JSON-formatted files to describe RESTful APIs, easier to use.
  • Swagger 2.0 (2014): Added support for YAML (YAML Ain't Markup Language), which is more human-readable, and allowed describing all aspects of an API in a single file.
  • OpenAPI Initiative (2015): Swagger format was donated and became OpenAPI version 2.
  • OpenAPI 3.0.0 (2017): Further improved structure, reusability, and added features like examples and callbacks.
  • Apigee currently supports OpenAPI versions 2 and 3.

3 API Proxies

1. What is a Proxy?

In Apigee a proxy acts as an intermediary between a requester and a provider. The requester sends a request to the proxy, and the proxy forwards it to the provider. The provider sends the response back to the proxy, which then fulfills the request to the requester. This means the requester doesn't need to know about the provider, and the provider doesn't need to communicate directly with the requester.

2. What is an API Proxy?

An API proxy is a facade layer that separates API consumers (applications) from backend services and resources. Instead of connecting directly to backend services, the API consumer connects to the API proxy. The API proxy then handles the required functionality by calling the backend services.

Key functions of an API proxy:

  • Decouples API consumers from backend services: This allows changes to backend systems without breaking applications that consume the APIs.
  • Implements consumer-facing APIs: These APIs can be designed specifically for app developers.
  • Shields app developers from backend complexity: The API proxy can handle concerns like authentication, authorization, data manipulation, and removing sensitive data, abstracting this complexity from app developers.
  • Allows backend changes without affecting apps: Modernizing backend services or moving them to the cloud can be done without impacting applications using the proxy APIs.

3. Apigee API Proxy

An Apigee API proxy:

  • Receives API requests based on specified hostname, scheme, and URL path combinations.
  • Adds security, scalability, and rate limiting to APIs by attaching policies to flows.
  • Can call multiple services, combine, and transform their responses during an API call.
  • Captures analytics for all calls, providing visibility into requests, responses, and backend services.

4. Proxy Endpoints and Target Endpoints

An Apigee API Proxy is split into two main parts:

  • Proxy Endpoint: This is the entry point for an API call, closer to the API consumer.
    • Traffic matching the protocol, host alias, and base URL is routed to this endpoint.
    • Parses and verifies inbound requests.
    • Policies are attached to rate-limit traffic and protect against invalid/malicious requests, often blocking them before they reach the backend.
    • Typically builds the API response to be sent back to the caller.
  • Target Endpoint: This handles the calling of the backend service, closer to the backend.
    • Specifies the URL of the backend service to call.
    • Builds API requests for the backend service.
    • Specifies connection and I/O timeouts for backend service calls.
    • Validates status codes and payloads of backend responses.
    • A proxy can have multiple target endpoints, but only one is called during an API request.
    • It's also possible to have zero target endpoints if the proxy handles the entire request/response.

5. Routing to an API Proxy

  • Environments: Each organization can have multiple environments (e.g., development, test, production) to control the API lifecycle.
  • Hostnames: The Apigee runtime can receive traffic using different hostnames (e.g., `api.example.org` for production, `test-api.example.org` for test).
  • Base Path: The first part of the URL following the hostname (e.g., `/orders/v1` in `https://api.example.org/orders/v1/...`). A base path is configured in a proxy endpoint, and only traffic starting with that base path is routed to that endpoint.
  • Deployment Consideration: When deploying a proxy revision to an environment, all proxy endpoint base paths must be unique within that environment. If multiple base paths match a request, the longer base path match will be chosen.

6. Variables

Apigee API proxies allow the use of variables to control proxy functionality.

  • Predefined Variables: Automatically set during an API call. Examples include `request.header.*`, `request.queryparam.*`, `request.verb`, `request.content`, `response.header.*`, `response.status.code`, `system.time`, and `target.url`.
  • Custom Variables: You can create your own variables to store information for building requests and responses, logging, or controlling processing flow.
  • `proxy.pathsuffix`: A commonly used predefined variable that is set to the part of the incoming URL following the base path (excluding query parameters). Along with the request verb, it typically indicates the operation being performed in a RESTful API.

7. Conditions

Conditions are similar to "if statements" in programming languages and must always evaluate to true or false. They allow dynamic handling during API calls.

  • Can use predefined and custom variables.
  • Can use string literals, numeric literals, 'true', 'false', and 'null'.
  • Can be chained using 'AND' and 'OR' operators.
  • Pattern Matching Operators:
    • `==`: Exact match.
    • `Matches`: Match with a wildcard (`*`).
    • `JavaRegex`: Allows use of regular expressions.
    • `MatchesPath`: Used for matching URL patterns. `/` matches a single segment, `/***` matches one or more segments.
  • All pattern matching operators are case-sensitive.

8. API Proxy Flows

As an API call is processed, it travels through a sequence of flows, and policies attached to these flows are executed.

Evaluation Order:

  • Proxy Endpoint Request: Request validation and parsing typically happen here.
  • Route Rules: Evaluated to determine which target endpoint, if any, should continue processing.
  • Target Endpoint Request: The request to be sent to the backend is generally built here.
  • Request sent to target, and response received.
  • Target Endpoint Response: Backend response parsing typically happens here.
  • Proxy Endpoint Response: The API response is built, and necessary information is logged.

Flow Structure within Each Section (PreFlow, Conditional Flows, PostFlow):

  • PreFlow: Policies in PreFlow are always evaluated first.
  • Conditional Flows: Contain zero or more flows, each with an optional condition. They are evaluated in order, and only policies in the first conditional flow with a true condition are executed. If no conditions are true, none are executed.
  • PostFlow: Policies in PostFlow are always evaluated after any conditional flows.

9. Policies

Policies are pre-built modules that implement specific, limited functions within an API proxy. They are configured using simple XML and typically do not require custom code, though some allow for custom code execution. A policy can be conditionally executed.

Policy Categories:

  • Traffic Management: SpikeArrest (rate limiting), Quota, Cache policies (Lookup/populate/invalidate cache, ResponseCache).
  • Security: VerifyAPIKey, OAuthV2 (generate, verify, refresh, invalidate tokens), BasicAuthentication, JWT/JWS (generate, verify, decode), SAML assertion (generate, validate), HMAC, CORS, JSON/XML Threat Protection, Regular Expression Threat Protection, Access Control.
  • Mediation: Assign Message (create, modify, remove HTTP message elements/variables), ExtractVariables, KeyValueMapOperations, RaiseFault, JSONtoXML, XMLtoJSON, XSL Transform, MessageValidation, OpenAPI Spec Validation (OASValidation).
  • Extension: JavaScript, JavaCallout (custom code), ServiceCallout (call external services/other proxies), FlowCallout (call shared flows).
  • Other: MessageLogging, DataCapture.

Attaching Policies: Policies are attached within flows as "steps." Each step can have an associated condition, meaning the policy is executed only if the condition is true. Steps within a flow are evaluated in order.

10. Route Rules

Route rules determine which target endpoint will be used for an API call, if any.

  • Evaluated in order after the proxy endpoint request flows complete.
  • The first matching route rule indicates the target endpoint.
  • Can have an optional condition; if no condition, the rule automatically matches.
  • Can specify no target endpoint, in which case processing continues with the proxy endpoint response.
  • Best Practice: The last route rule should have no condition to act as a default.

11. Target Servers

Target servers are an Apigee feature used to avoid hardcoding backend URLs in proxy configurations, especially when promoting proxies across different environments (e.g., test to production).

  • You create a target server configuration with the same name in each environment.
  • The hostname, port, and any certificate information can be different for each environment, but the target server name remains consistent.
  • In the proxy's 'HTTPTargetConnection', you reference the target server name within a `Load Balancer` element instead of a hardcoded URL. The actual hostname and port are then taken from the target server configuration specific to the environment where the proxy is deployed.
  • This allows the same proxy code to be used in all environments without modifications.

4 API Products

This document summarizes the main concepts and information regarding API products, API keys, and REST API responses.

API Products

  • Definition: An API product is a bundle of API operations designed for use by app developers. It acts as a mechanism to package, control access to, and manage service levels for APIs.
  • Purpose:
    • Packaging: Grouping relevant API operations together (e.g., product catalog, shopping cart, order completion).
    • Access Control: Granting or restricting access to APIs at a product level (e.g., blocking public access, tracking app usage).
    • Service Levels: Providing different levels of access or service for the same APIs (e.g., read-only/read-write, internal/public, free/standard/premium).
  • Custom Attributes: These are name/value pairs associated with an API product (or individual operations/proxies). When an app uses an API key, these attributes are populated as variables, allowing dynamic control of API proxy functionality (e.g., `accessType: internal/external`, `service-plan: free/standard/premium`).
  • Product Mindset vs. Project Mindset:
    • Project: Focuses on deliverables, delivery dates, and has a completion date. Requirements are often locked down upfront.
    • Product: Focuses on outcomes, adapts to market needs, and does not have a completion date. This approach leads to better adoption and customer satisfaction.
  • Design Strategies:
    • API products should be designed based on the needs of app developers, not internal implementation teams.
    • APIs and API operations from different teams can be combined into a single API product.
    • Multiple API products can provide different levels of access to the same underlying APIs (e.g., an internal read-write product, an internal read-only product, and an external limited-access product).

API Keys

  • Role: An API key is typically the 'consumer key' that identifies an app and allows it to access protected API operations.
  • Protection: API keys are used to protect APIs from unauthorized access. Requests without a valid API key (or one that doesn't have access to the specific API/operation) can be rejected.
  • VerifyAPIKey Policy:
    • This policy is almost always attached early in the proxy endpoint request preflow.
    • It validates the API key.
    • If the key is valid, variables are created for associated app, developer, and API product attributes (including custom attributes). These variables can then be used to control the API proxy's behavior dynamically.
    • If the API key is invalid or doesn't grant access, the policy raises a fault, stopping processing and returning an error message.
    • The policy typically looks for the API key in a query parameter or HTTP header.

REST API Responses

HTTP Status Codes:

  • Purpose: Primary signals for success or failure of an operation. They have specific, defined meanings and should be used consistently across all APIs.
  • Common Codes:
    • 200 OK: Request successful.
    • 400 Bad Request: Client's request didn't satisfy requirements (fixable by client).
    • 401 Unauthorized: Authentication required, user not authenticated.
    • 403 Forbidden: User authenticated but not authorized to perform the request.
    • 404 Not Found: Resource specified in URL doesn't exist or is inaccessible.
    • 429 Too Many Requests: Request rejected due to too many recent requests (used by Apigee for spike arrest and quota policies).
    • 500 Server Error: Unexpected error on the server side (caller might retry later).
  • Other Useful Codes:
    • 201 Created: Resource successfully created.
    • 204 No Content: Success, but no content in response payload.
    • 304 Not Modified: Resource version caller has is most up-to-date.
    • 405 Method Not Allowed: Valid resource, but requested HTTP verb not allowed.
    • 406 Not Acceptable: Requested response format not supported.
    • 409 Conflict: Request cannot be completed due to a conflict (e.g., resource changed after original read).
    • 503 Service Unavailable: Service temporarily unable to handle request (e.g., maintenance).
  • Status Code Ranges:
    • 1XX - Informational: Processing not complete.
    • 2XX - Success: Request received, understood, accepted.
    • 3XX - Redirection: Client must take further action.
    • 4XX - Client Error: Error due to client's request (client's fault).
    • 5XX - Server Error: Error on the server (not client's fault).
  • Important Note: Do not automatically pass backend status codes directly to the client if they are misleading or expose sensitive internal information. Always consider the meaning of the status code being returned to the app developer.

Response Body Content:

  • Bad Responses:
    • Lack of context for how to fix a client-side error (e.g., 400 Bad Request without details).
    • Information leakage (e.g., returning 403 Forbidden when a 404 Not Found would prevent revealing resource existence).
    • Leakage of internal server details (e.g., backend system names, internal IP addresses in a 500 Server Error).
  • Good Responses:
    • Provide clear messages explaining the error and how to fix it (for client errors).
    • Avoid leaking sensitive information about the backend or resource existence if the user is not authorized.
    • Include a correlationId for server errors to help with debugging without exposing internal details.

Pagination:

  • Purpose: To return a subset of ordered results at a time, especially for searches or queries with a large or unknown number of results.
  • Types:
    • Offset and Limit: Works well for UI use cases. 'offset' indicates the starting resource, 'limit' the number of resources to return.
    • Value-based Paging: Used for traversing all results consistently. A "last" parameter (e.g., timestamp, ID) is returned with the last retrieved resource, used in the next call to retrieve subsequent resources.

Recap

Key Apigee Elements & Their Relation

  1. Environment Group:
    • What it is: A logical grouping of one or more Apigee Environments. It defines the public-facing domain names (hosts) through which your APIs are accessed.
    • Relation: Contains Environments. Provides the top-level URL for your APIs.
    • Example: `apigee-org-prod.net` (domain)
    • Corresponding Elements:
      • Domain: The public hostname for API calls.
  2. Environment:
    • What it is: A runtime execution context for API Proxies and Shared Flows. Typically maps to development stages (e.g., dev, test, prod). Each environment is isolated.
    • Relation: Lives within an Environment Group. API Proxies are deployed to an Environment.
    • Example: `prod` (environment within `apigee-org-prod.net`)
    • Corresponding Elements:
      • Domain: Inherits from the Environment Group (e.g., `apigee-org-prod.net`).
  3. API Proxy:
    • What it is: The Apigee-managed interface that clients consume. It acts as a facade in front of your backend services, providing security, traffic management, mediation, etc.
    • Relation: Contains Proxy Endpoints, Target Endpoints, and Policies. Has Revisions. Deployed to an Environment.
    • Corresponding Elements:
      • Example: `Products-API`
      • Paths: Defined in Proxy Endpoints.
  4. Revisions:
    • What it is: A numbered snapshot of an API Proxy's configuration. Every time you save changes to an API Proxy, a new revision is created.
    • Relation: An API Proxy has multiple revisions. Only one revision of a proxy can be deployed to an environment at a time (per base path).
    • Example: `Products-API Revision 1`, `Products-API Revision 2`
    • Corresponding Elements:
      • State: Deployed/Undeployed.
  5. Proxy Endpoint:
    • What it is: Defines the client-facing configuration of an API Proxy. This includes the base path (the public URL clients use) and any policies applied before the request is sent to the backend.
    • Relation: Part of an API Proxy. Defines the public interface. Policies are attached to its flows.
    • Example: For Products-API, a Proxy Endpoint with `/v1/products` base path.
    • Corresponding Elements:
      • Paths: `/v1/products`
      • Codes: HTTP status codes for responses.

    You can use multiple ProxyEndpoints in a single Apigee API proxy to expose the same backend logic through different interfaces, each with its own network settings or security rules.

  6. Policies:
    • What it is: Functional building blocks that enable specific API management capabilities (e.g., API key verification, quota, JSON-to-XML transformation, traffic spike arrest).
    • Relation: Attached to specific "flows" (request/response paths) within Proxy Endpoints or Target Endpoints.
    • Example: `VerifyAPIKeyPolicy`, `QuotaPolicy`, `ResponseCachePolicy`
    • Corresponding Elements:
      • Codes: Can generate HTTP error codes (e.g., 401 for unauthorized, 429 for quota exceeded).
  7. API Product:
    • What it is: A bundle of API Proxies (or specific paths/operations within them) that you make available to app developers. It's used to control access (via API keys), enforce quotas, and define monetization strategies.
    • Relation: Contains references to API Proxies (or their resources). Developers obtain API Keys linked to API Products.
    • Example: `Premium-Products-Access` (might include Products-API `/v1/products` GET and POST methods)
    • Corresponding Elements:
      • Paths: The specific paths/methods included from proxies.
      • Codes: Can lead to 403 (Forbidden) if API Key lacks product access.

5 Authentication, Authorization, and OAuth

This document summarizes key concepts and information related to API security, with a focus on OAuth.

API Security Concerns

  • Data Breaches: Many data breaches involve inadequately secured applications and APIs.
  • Importance of Security: As APIs become crucial for companies, securing them is equally important to prevent data breaches.
  • Consider Security for All APIs: Even if an API doesn't seem to contain sensitive data, thoughtful consideration of security requirements is always necessary.

API Exposure Types

  • Public APIs:
    • Available and discoverable on the internet.
    • Typically contain sensitive data.
    • Require authentication and data security to protect against easy access for bad actors.
  • Private APIs (Internal APIs):
    • Not exposed on the internet.
    • While the attack vector is smaller, sensitive APIs should still be secured to protect user data from internal breaches.
    • Any sensitive APIs accessible outside the firewall must be fully authenticated and encrypted.

Security Considerations

  • Infrastructure and Network Security: The underlying infrastructure of an API management platform must be secure. Google Cloud and Apigee API Platform are designed with security in mind and offer tools like Cloud Armor (web application firewall) and reCAPTCHA (bot protection).
  • Traffic Management: Source IP: Use Access Control policies to allow or deny traffic based on specific IP addresses or ranges.
  • Injection and Data-Based Attacks:
    • APIs often accept data payloads to be saved in backend databases, making them vulnerable to injection attacks (sending untrusted data to alter service execution).
    • Apigee's API proxy acts as a security layer to inspect incoming data before it reaches backends.
    • Regular Expression Protection policy: Configures regular expressions to detect dangerous content in requests (payload, headers, query parameters) and reject them.
    • Still, adhere to best practices like parameterized queries and escaping/encoding data in your backend services.
    • The RegularExpression Protection policy can detect these types of attacks.
  • Application-Level Denial of Service (DoS):
    • Large or malformed XML or JSON structures can overwhelm services, causing excessive memory use and reduced API capacity.
    • JSON Threat Protection policy & XML Threat Protection policy: Detect dangerous payloads without parsing them, protecting both the proxy and backend.
  • Data Confidentiality:
    • Regulations and privacy concerns necessitate limiting access to user data.
    • Role-based access control: Limits internal user access to production environments, including tracing of live production traffic, by assigning specific permissions to roles.
    • Data masking and private variables: Prevent sensitive data from appearing in trace sessions, even for support teams debugging issues.

Identity, Authentication, and Authorization

These are distinct concepts in API security:

  • Identity Tracking: Determining the identity of the app or user for a request, similar to checking if someone has a ticket. It identifies the caller but doesn't prove who they are.
    • Apps: Generally identified using API keys.
    • Users: Identified by username or account number.
  • Authentication (AuthN): The process of validating credentials to prove identity. It's like asking "Who are you? Prove it."
    • App credentials: Consumer key and secret.
    • User credentials: Often username and password.
    • Machine credentials: TLS certificate.
    • Protection: All authentication traffic must be sent using Transport Layer Security (TLS).
    • Standards: Active Directory (AD)/LDAP, OpenID Connect.
  • Authorization (AuthZ): Determining what an authenticated app or user is allowed to do. It's like saying "Here's what you are allowed to do."
    • Only makes sense after successful authentication.
    • Proof of authorization: Often via tokens.
    • Standards: OAuth 2.0, SAML 2.0.

Common Patterns of Usage in API Requests:

  • API Key: Can be passed as a query parameter (less recommended due to logging) or, preferably, in a header (e.g., `ApiKey: 45c78ece5b77647854a`).
  • Basic Authentication: Used for user or app credentials (username and password or consumer key and secret) in the `Authorization` header (e.g., `Authorization: Basic YmFkOnBhc3N3b3Jk`). The value is base64 encoded, which is not encryption and relies on TLS for protection.
  • Access Token: Used for authorization, generally passed as a bearer token in the `Authorization` header (e.g., `Authorization: Bearer 4WCAchNNtVyK8JsACI1HP7ml`).
  • Token Format: There are two common types of access tokens:
    • Opaque Tokens: These are random strings of characters with no meaning to the client (e.g., `4WCAchNNtVyK8JsACI1HP7ml`). Only the authorization server (like Apigee) can look up this token in its database to validate it.
    • JSON Web Tokens (JWTs): These tokens are digitally signed, not encrypted. The signature guarantees that the token was issued by a trusted server and that its contents have not been altered. Anyone can decode a JWT to see the information inside, but only the server can create a valid signature.

OAuth (Open Authorization) Framework

OAuth is an authorization framework for REST APIs that allows users or clients to grant access to server resources to another entity without sharing their credentials. The current industry standard is OAuth 2.0, supported by Apigee.

  • Access Tokens:
    • Issued by an authorization server (e.g., Apigee acting as one).
    • Allow limited access to specific resources for a specified period of time.
    • Can be revoked by the user or the issuing server if compromised.
    • In Apigee, they are opaque strings (random characters) with no encoded meaning; all information is stored on the authorization server.
    • Should be passed as Bearer tokens in the 'Authorization' header.
    • Obtaining an Access Token: Requires the application to authenticate (client ID + secret), and often the resource owner (user) to authenticate, along with an optional scope.
  • Refresh Tokens:
    • Provide a limited right to reauthorize granted access by obtaining new access tokens without requiring user credentials.
    • Used when the previous access token expires.
    • Also opaque strings in Apigee.
    • Obtaining a new Access Token with a Refresh Token: The app provides its client ID and secret, the refresh token, and an optional scope. The new access token generally has a short lifespan (e.g., five minutes).
  • Other Terms:
    • Grant types: Authentication usage scenarios supported by OAuth 2.0 (four specified types).
    • Client IDs and client secrets: Used to identify and authenticate apps (Apigee's consumer key and consumer secret).
    • Scopes: Limit the access for a given token, granting permission only for necessary operations. The authorization server defines scope names and access levels. API proxies can block requests based on scope.
    • TLS (Transport Layer Security): All OAuth 2.0 traffic (including API requests with tokens) must be encrypted via TLS (successor to SSL) to protect sensitive information.

How a User Controls Scopes

A user's control over scopes happens at two main points:

  1. During Initial Authorization: When an application first asks for permission (e.g., "This app wants to view your contacts and manage your calendar"), the user is presented with a consent screen. At this point, they can agree to grant the requested scopes. Some platforms may even allow users to uncheck certain permissions before proceeding.
  2. After Authorization (In Account Settings): After granting access, a user can typically go to their account management page (e.g., Google's "Security" settings). From there, they can see all third-party applications they've authorized and can choose to revoke access entirely. Revoking access invalidates the application's token and removes all its permissions (scopes).

An application cannot get new scopes without re-initiating the authorization process and asking the user for their consent again.

Terminology: Types of Apps

  • Confidential Apps

    These are applications that run on a secure server where the code is not exposed to the end-user. Because of this, they can safely store a client secret.

    Example: A backend server for an e-commerce website. When it needs to call a payment gateway API like Stripe, the backend server can securely use its client ID and client secret because those credentials live on the server, completely hidden from users' browsers.

  • Public Apps

    These applications run in an insecure environment, typically the user's device or browser, where they cannot protect a client secret.

    Example: A single-page web application (like a React or Angular app) or a native mobile app. The application's code can be inspected by the user, so embedding a client secret would be like taping your house key to the front door. These apps use OAuth flows (like Authorization Code with PKCE) that don't require a client secret.

  • Trusted Apps

    These are first-party applications that are so trusted by the user and the organization that they can be given the user's password directly.

    Example: Your official mobile banking app. You enter your banking username and password directly into the app because you trust that it is built and secured by your bank. This is an exception, and this level of trust is rare.

  • Untrusted Apps

    This refers to any third-party application that should never handle a user's password directly. This is the primary problem that OAuth was created to solve.

    Example: A third-party app like Canva that asks to access your Google Drive photos. You are redirected to a Google sign-in page to grant permission. Canva never sees your Google password; it only receives an access token that gives it permission to access your photos and nothing else.

OAuth Grant Types

OAuth grant types are simply the different methods an application can use to get an access token. Each method, or "flow," is designed for a different type of application or scenario.

In Apigee, you configure the OAuthV2 policy to handle these different flows.

  1. Authorization Code

    This is the most common and secure method, used by web and mobile apps where a user logs in.

    • It's like clicking "Log in with Google" on a website. The site redirects you to Google to sign in, and you then grant it permission. The site gets a temporary code, which it exchanges behind the scenes for an access token.

    Use Case: A user authorizing a third-party application (like a travel booking site) to access their calendar or contacts.

  2. Client Credentials

    This method is for machine-to-machine (M2M) communication where there is no user.

    • A backend server needs to get data from another internal service. The server authenticates using its own credentials (a client ID and secret) to get a token.

    Use Case: A nightly data-processing script that needs to call an internal reporting API.

Create the OAuth 2.0 Token Proxy

First, you need an API proxy dedicated to issuing access tokens. This proxy will expose the standard `/token` endpoint.

  • Create a new API proxy (e.g., named `oauth-v1`). This proxy won't have a backend target; its job is to run OAuth policies and generate tokens.
  • Attach the OAuthV2 Policy: Inside this proxy, you will create flows that use the OAuthV2 policy. You will configure a separate flow for each grant type you want to support.
    • Client Credentials Flow: Configure the policy with `GenerateAccessToken` and `client_credentials`.
    • Authorization Code Flow: This is more complex and requires multiple policy operations (GenerateAuthorizationCode, GenerateAccessToken), typically involving redirects to a login page.

Set Up the API Product and App

OAuth 2.0 relies on client credentials (client ID and secret) which are managed through API products and developer apps.

  • Create an API Product: Create a product (e.g., Retail Product) and associate it with the protected API proxies (e.g., your `/products` and `/orders` APIs), but not with the `oauth-v1` token proxy.
  • Register a Developer App: Create a developer and then register an app for them. When you add your Retail Product to this app, Apigee will automatically generate a Consumer Key (Client ID) and Consumer Secret. These are the credentials the client application will use to call your token endpoint.

Secure Your Resource APIs

Finally, you need to enforce token validation on the APIs you want to protect.

  • Go to your resource proxies (e.g., the proxy that handles `/products`).
  • Attach the VerifyAccessToken Policy: In the PreFlow of the request, add an OAuthV2 policy configured with the `VerifyAccessToken`. This policy will check for a valid `Authorization: Bearer {token}` header. If the token is missing, invalid, or expired, the policy will reject the request.

Token creation and token validation

They are connected through the API Product.

The VerifyAccessToken policy on your resource API automatically checks if the token presented is valid for the specific API Product that protects that resource.

Token Creation: A client application sends its Client ID and Client Secret to your `/token` endpoint. Apigee knows these credentials belong to a specific Developer App. That app, in turn, is authorized for one or more API Products. When Apigee creates the token, it internally links it to that API Product.

  1. Token Validation: The client then sends that token to your resource API (e.g., `/products`). Your VerifyAccessToken policy on this proxy does the following check:
    • "Is this a valid, unexpired token?"
    • "Is the API I am protecting (`/products`) part of the API Product this token was issued for?"

If the answer to both questions is yes, the request is allowed. If the token was issued for a different API Product that doesn't include the `/products` API, the policy will reject the request.

6 Content, Transport, and Platform Security in Apigee

This document summarizes key concepts and information regarding content, transport, and platform security when developing APIs with Apigee.

Content-Based Attacks and Protection

Content-based attacks exploit malformed API requests to cause issues with APIs and backend services. These attacks can involve manipulating text fields to leak or destroy data, or crafting JSON/XML payloads to disrupt parsing and cause application-level denial of service. Legacy services are often vulnerable as they may not have been designed with external traffic in mind.

Protection:

It's crucial to block these attacks at the API proxy level before data reaches the backend. Apigee offers policies to help mitigate these threats.

Input Validation Tips:
  • Use the ExtractVariables policy with JSONPath/XPath to extract fields from payloads.
  • Validate required fields, field interactions, and regular expression patterns using proxy conditions or JavaScript policies.
  • Rewrite validation error messages from the backend to maintain a consistent format.
Malicious Input Checks:
  • Block dangerous strings like SQL injection code in the proxy, even if the backend also performs checks.
  • Still, adhere to best practices like parameterized queries and escaping/encoding data in your backend services.
  • The RegularExpression Protection policy can detect these types of attacks.
Regular Expression Protection Policy:
  • Searches for blocked patterns in various locations (URI, payload, headers, query parameters, variables).
  • If a specified regular expression matches, the message is considered a threat and rejected.
Content-Type Header:
  • Specifies the format of incoming payloads (e.g., 'application/json', 'application/xml').
  • Correctly setting the 'Content-Type' header is essential for JSON or XML parsing policies to function. If incorrect or missing, parsing attempts by policies like ExtractVariables might be silently skipped.
  • You can use a series of policies to handle 'Content-Type' header scenarios, such as raising a fault for invalid types or setting a default if it's null.
Parsing Attacks and Threat Protection Policies:
  • XML and JSON payloads can be crafted to overwhelm parsers, leading to denial of service.
  • JSONThreatProtection and XMLThreatProtection policies use string-based evaluation instead of parsing the payload. This allows them to detect payloads exceeding configured limits and reject requests before a parser is engaged.
  • These policies should always be run before any JSON/XML parsing.
JSONThreat Protection Policy:
  • Allows specifying limits on JSON structures and fields (e.g., `ArrayElementCount`, `ContainerDepth`, `ObjectEntryCount`, `ObjectEntryNameLength`, `StringValueLength`).
  • These limits are checked without using a parser.
  • The first exceeded limit results in an error. It's important to set limits that allow legitimate requests to pass.
XMLThreat Protection Policy:
  • Functions similarly to the JSONThreat Protection policy but with specific configuration elements for XML (e.g., `NameLimits`, `StructureLimits`, `ValueLimits`) due to the greater complexity of the XML format.

Transport Security

Transport Layer Security (TLS), the successor to Secure Sockets Layer (SSL), establishes encrypted HTTP links (HTTPS) between clients and servers. Certificates are used to prove identity. TLS is recommended for all API traffic and is a firm requirement for OAuth traffic.

  • One-way TLS (Server Validation):
    • Standard web HTTPS.
    • The server presents a certificate to prove its identity; the client validates it using a truststore.
    • The client does not present a certificate.
  • Two-way TLS (Mutual Authentication):
    • Recommended for securing the link between Apigee and backend services.
    • Both the client and server present and validate each other's certificates.
    • This provides stronger authentication, ensuring requests to the backend are from a legitimate source.
  • Keystores and Truststores:
    • Keystore: Stores certificates to be presented to a remote server and the associated private key to prove identity.
    • Truststore: Stores certificates to compare with remote certificates, allowing the proxy to verify the identity of the communicating participant.
  • On Apigee, keystores and truststores are used for secure connections.
  • Configuring TLS for Apigee:
    • Apigee: TLS is configured on the load balancer in the Google Cloud project.
    • Apigee Backend: TLS is configured in the TargetEndpoint of the proxy or in a TargetServer configuration.
    • A truststore is used for both one-way and two-way TLS to validate backend certificates.
    • For two-way TLS, an Apigee keystore is used to store private keys and certificates presented to the backend.
  • References for Keystores and Truststores:
    • Keystores and truststores can be specified by name or by using a reference (a variable containing the name).
    • Using references allows for key and certificate rotation without downtime, as a new keystore/truststore with updated certificates can be created, and the reference can be updated on the fly.
  • Access Control Policy:
    • An additional method to restrict incoming connections.
    • Allows for IP allow/deny lists for ranges of IPs.
    • Rules are evaluated in order, and the `noRuleMatchAction` specifies the default behavior if no rules match.
    • The ClientIPVariable can be used to specify the IP address to check, which is useful when Apigee is behind a load balancer (often looking at the `X-Forwarded-For` header).

Apigee Platform Security

Platform security focuses on protecting internal access within the Apigee environment.

  • Data Masking:
    • The Apigee trace tool allows users to see data during API calls, including sensitive information.
    • Data masking blocks configured fields from being viewed in live traces or trace files, replacing sensitive data with asterisks.
    • Configured using the Debug Mask Apigee API at the environment level.
    • It's crucial to mask sensitive data like passwords and credit card numbers.
  • Debug Mask Singleton:
    • A single object in an Apigee environment that can be patched to define data masks.
    • The payload for the Debug Mask specifies variables and JSONPath/XPath expressions for requests or responses to be masked.
  • Private Variables:
    • Variables prefixed with `private.` (e.g., `private.password`).
    • Values of private variables are automatically masked in traces when created or accessed.
    • Important: If a private variable is assigned to another non-private variable, the value will become visible in the new variable.
  • Key Value Maps (KVMs):
    • Used to store non-expiring, configuration data that is retrieved at runtime, similar to property files.
    • Keys and values are strings.
    • Typically environment-scoped.
    • Ideal for environment-specific configuration.
    • Can be created/deleted using the Apigee API or console.
    • Entries can be read/updated/deleted using the Apigee API or the KeyValueMapOperations policy.
    • KVMs are encrypted, making them safe for sensitive data like backend credentials.
    • Values can only be retrieved into private variables.
  • KeyValueMapOperations Policy:
    • Used within a proxy to insert, update, and retrieve KVM entries.
    • `MapName` specifies the KVM to access.
    • `Scope` defaults to 'environment'.
    • Uses 'Get' for retrieval and 'Put' for adding/updating entries.
    • One or more Parameter fields are used to define the key.
    • `ExpiryTimeInSecs` specifies how long to cache the entry.
  • Property Sets:
    • Custom collections of key/value pairs stored in `.properties` files.
    • Used for static, non-expiring data that should not be hardcoded.
    • Can be scoped to an API proxy or all API proxies in an environment.
    • Updating an API proxy-scoped property set requires redeployment.
    • Accessed using flow variables (e.g., `propertyset.property_set_name.property_name`).
  • Comparing KVMs and Property Sets:
    • KVMs: Values are encrypted, can be updated dynamically by API proxies, and changes to proxy-scoped KVMs do not require redeployment. Best for sensitive or dynamic configuration.
    • Property Sets: Values are stored in plaintext, cannot be updated dynamically by API proxies (except for environment-scoped which update without redeployment), and API proxy-scoped changes require redeployment. Simpler for static, non-sensitive configuration data and are often checked into code repositories.

7 API Mediation

This document outlines key concepts and information related to API mediation using Apigee, covering various policies and best practices.

1. Introduction to API Mediation

API mediation in Apigee involves transforming requests and responses and calling multiple services during a single API call to simplify app development and leverage backend services more effectively. This often involves the Apigee API proxy acting as an abstraction or facade layer for one or more backend services.

2. JSON, XML, and SOAP Support

Apigee supports various data formats, including JSON, XML, and SOAP.

  • Content-Type Header (Reminder): The 'Content-Type' header is crucial as it specifies the format of payloads. If it's not correctly set (e.g., `application/json` for JSON), parsing by policies will be silently skipped. Always validate or set this header.
  • Message Validation Policy: This policy validates messages and raises a fault if they don't conform to specified requirements. It can be used to:
    • Validate XML against XSD (XML Schema Definition) files.
    • Validate SOAP against WSDL (Web Services Description Language) files.
    • Confirm that JSON or XML is well-formed.
  • XSD and WSDL files must be stored in the proxy resources.

3. Mediation Patterns

Format Conversion

Backend or third-party services often use different content types or payload formats than your API. Even if formats are the same (e.g., JSON), the payload structure might need rewriting.

  • ExtractVariables Policy: This policy extracts information from messages or variables and stores the results in variables for later use in your proxy.
    • Variables can be extracted from headers, query parameters, URI paths, or any existing variable.
    • It supports multiple patterns for a particular location, evaluated in order, with the first match being used.
    • For URI paths, variable matching cannot span multiple path segments.
    • You can use indexing (e.g., `type.1`) to select specific occurrences of multiple headers or query parameters with the same name.
    • Set `IgnoreUnresolvedVariables` to 'true' if some variables are optional; otherwise, an error will be raised if a match isn't found.
    • Can extract data from JSON or XML payloads using JSONPath or XPath respectively.
  • Assign Message Policy: This policy creates variables and HTTP messages.
    • Variables can be created by reference (`Ref`) or by value (`Value`).
    • It can modify existing messages or create new ones using the `createNew` attribute of the `AssignTo` element.
    • The `Set` element allows you to set fields in a message (verb, payload, headers, query/form parameters, status codes, request path).
    • The `Remove` element allows you to remove fields from a message (specific headers or all query parameters).
    • Operations within the policy are executed in order.
  • Templates: Allow dynamic string substitution in certain policy and `TargetEndpoint` elements.
    • They use message template functions and variables (e.g., `timeFormatUTCMs`, `encodeBase64`, `jsonPath`).
    • Many functions are available for string manipulation, hash calculation, random number generation, etc.

3.2. Orchestration

This pattern involves calling multiple services during a single API request/response.

  • Combining multiple steps into a single operation can improve developer experience and reduce latency.
  • Adding data from third-party services can increase the utility of an API call.
  • ServiceCallout Policy: Calls out to an external service during the request and response flows of a proxy.
    • Uses separate request and response message variables from the main proxy request and response.
    • The request can be specified by variable or built within the policy.
    • The response is specified as a variable that will be populated after the service call.
    • You can specify a `Timeout` in milliseconds; if reached, the policy throws an error.
    • The target for a service callout is configured similarly to a TargetEndpoint target.
  • Proxy Chaining: Allows calling proxies deployed in the same organization and environment.
    • Uses `LocalTargetConnection` instead of `HTTPTargetConnection`.
    • The called proxy can be specified by name and endpoint or by path.
    • Calls using `Local TargetConnection` stay within the gateway, avoiding network hops and additional network connections.
    • Can be used for `TargetEndpoints` and `ServiceCallouts`.
    • A strategy for avoiding duplicate code, though shared flows are a more powerful method.

4. Custom Code

Apigee provides policies to write custom code within proxies.

  • Supported Languages: JavaScript, Java, and Python (Jython 2.5.2).
  • Preference: Prefer configuration-based policies (e.g., `AssignMessage`, `ExtractVariables`) as they are optimized for efficiency and perform better.
  • When to Use Custom Code:
    • When needed functionality is not supported by configuration-based policies (e.g., iterating through and modifying complex payloads).
    • When configuration-based implementation would be too complex or confusing.

4.1. JavaScript Policy

  • Uses custom compiled server-side JavaScript.
  • References JavaScript code files stored in the proxy resources (e.g., `IncludeURL` for libraries, `ResourceURL` for the main file).
  • Allows getting, setting, and removing flow variables using `context.getVariable()`, `context.setVariable()`, and `context.removeVariable()`.
  • Useful for looping through and modifying JSON and XML payloads (e.g., parsing `request.content`, manipulating the object, and stringifying it back).
  • Can call multiple HTTP services in parallel using the `httpClient` object, which is an advantage over `ServiceCallout` for reducing latency.
  • `timeLimit` attribute specifies a timeout in milliseconds; if reached, execution stops and a fault is raised.
  • The `print()` function logs to the console in the Trace Tool for debugging.
  • For simple cases, code can be directly inserted into the `Source` element of the policy configuration file.

4.2. JavaCallout Policy

  • Allows using custom Java code in proxies, offering very good performance.
  • Requires specifying the class to run within the configured Java JAR file (`ClassName` and `ResourceURL`).
  • Can specify a list of name-value properties that can be used within the Java code.
  • Disadvantages:
    • Harder to maintain and debug as code is stored in compiled JAR files and needs to be compiled outside Apigee.
    • Java code is not visible in trace and cannot log to the trace tool console.
    • Most system calls are not allowed (e.g., network socket I/O, process/CPU/memory info, file system reads/writes).
  • Can access, create, update, and remove flow variables and message elements.

4.3. Choosing Between Code Policies

  • First Choice: Configuration-based policies (`ExtractVariables`, `Assign Message`).
  • Second Choice (Most Custom Code Use Cases): JavaScript policy. It's pre-compiled, performs well, and is easy for parsing/modifying JSON/XML, especially with arrays, or when code is more intuitive.
  • Third Choice: JavaCallout when performance is the absolute highest priority (e.g., heavy cryptography) or when existing Java libraries are required.
  • Avoid: Python policy, as it's interpreted at runtime and can be slow, especially under heavy load.

5. Shared Flows

Shared flows provide a powerful method for sharing reusable logic between API proxies.

  • Definition: A single flow of reusable logic containing policies, policy conditions, and resources.
  • Invocation: Cannot be invoked directly; must be called from a proxy or another shared flow.
  • Context: Executes within the context of the hosting proxy's flow, having read and write access to the proxy's flow variables (no need for message building/parsing).
  • Deployment: Hosted in an organization and deployed to an environment, similar to proxies. Can only be used by proxies or shared flows deployed to the same org/environment.
  • Benefits over Proxy Chaining: Generally better for sharing reusable logic because they execute within the hosting proxy's flow context.
  • FlowCallout Policy: Invokes a shared flow.
    • Can be used in proxies or other shared flows.
    • The shared flow must be deployed before the proxy or shared flow using it can be deployed.
    • The `SharedFlowBundle` element specifies the name of the shared flow to call.
    • Parameter elements can be added to create variables only visible during shared flow execution.
    • When tracing the proxy, you have full access to trace information for the shared flow steps.

5.1. Why Use Shared Flows?

  • Code Reuse: Easier and faster to write and debug a series of steps once, ensuring consistent task execution across proxies.
  • Maintenance: Easier to maintain code in one place rather than in multiple proxies. Changes can be applied centrally.

8 API Traffic Management

This document summarizes key concepts and information regarding API traffic management using Apigee, covering rate limiting (Spike Arrest and Quota policies) and caching (General-Purpose and Response Caching).

1. API Traffic Management Overview

API traffic management involves controlling and optimizing the flow of requests to your APIs. Apigee offers features to:

  • Control Traffic Rate: Reject excessive API traffic to protect backend services.
  • Limit Requests: Set specific limits on the number of requests allowed over a defined period.
  • Reduce Unnecessary Traffic: Use caching to minimize calls to backend services, improving performance and reducing costs.

2. Rate Limiting

Rate limiting is the process of rejecting excess traffic to your API and backend services. Apigee provides two built-in types: Spike Arrest and Quota.

2.1. Spike Arrest

The Spike Arrest policy protects backend services from sudden, large increases (spikes) in API traffic by limiting the allowed rate of requests. This is primarily a technical solution.

How it Works:
  • Does not use a counter: Instead of counting requests, it keeps track of the time the last matching request was allowed.
  • Rejects requests: If a message processor allowed traffic for the same identifier too recently, the current request is rejected.
  • Status Code: Rejected traffic returns a `429 Too Many Requests` status code.
  • Identifier: Groups traffic that will be rate-limited together (e.g., by client ID). If no identifier is specified, it applies to all traffic.
  • Rate Element: Specifies how often traffic is allowed, either "per minute" (`pm`) or "per second" (`ps`).
    • `10ps` means 1 request per 100 milliseconds (10 requests per 1000ms).
    • `30pm` means 1 request per 2 seconds (30 requests per 60 seconds).
  • Token Bucket Algorithm: Spike Arrest is implemented using this algorithm to allow for small bursts of traffic without immediate rejection, while still maintaining the overall specified rate.
    • A "bucket" is filled with "tokens" at a constant rate (e.g., every 100ms for `10ps`).
    • When a request is received, a token is consumed to allow the request. If no tokens are available, the request is rejected.
    • The bucket size is calculated as $\max(\text{rate}/10,1)$. For `10ps`, the bucket size is 1. For `600pm` (equivalent to `10ps`), the bucket size is 60, allowing a burst of 60 requests.

2.2. Quota

The Quota policy limits the number of requests allowed for a specific entity over a specified period. This solves a business problem.

How it Works:
  • Uses a counter: Tracks the number of accepted requests.
  • Rejects requests: Once the defined limit is reached, requests are rejected until the time period elapses and the counter resets to zero.
  • Does not control rate: Quotas do not protect against traffic spikes (e.g., all allowed requests for a month could be used in the first minute).
  • Status Code: Returns `429 Too Many Requests` for rejected requests.
  • Uniquely Identifies Counter: A combination of the proxy, policy, and Identifier value uniquely specifies a quota counter.
  • Configuration Elements:
    • Allow: Number of allowed requests per time period.
    • Interval: Number of time units in the period.
    • TimeUnit: Unit of time (minute, hour, day, week, month).
  • API Product Quota Settings: 'Allow', 'Interval', and 'TimeUnit' can be dynamically set using variables populated from API product quota settings, enabling different service levels (e.g., Basic vs. Premium). The `UseQuota ConfigInAPIProduct` element simplifies this configuration.
  • Quota Types: Specifies when the quota time period starts and resets.
    • Default: Starts at the beginning of the next GMT minute/hour/day/etc. and resets every interval.
    • Calendar: Explicit start time (`StartTime` element).
    • Flexi: Begins when the first message is received by the policy.
    • Rolling Window: Counts requests over a time period ending at the current request time (e.g., 100 requests in the past 2 hours).
  • Distributed Quotas:
    • `Distributed=false` (default): Separate counter for each message processor.
    • `Distributed=true`: A shared counter is used across all message processors (more common).
  • Synchronous vs. Asynchronous (with Distributed=true):
    • `Synchronous=true`: Counter updates immediately. Can introduce significant latencies, especially under load. Less likely to allow extra requests.
    • `Synchronous=false`: Counter updates periodically. May allow a few extra requests due to synchronization frequency. Recommended for most use cases if tolerating extra requests.
    • Asynchronous Configuration: `SyncMessageCount` (syncs after N requests) and `SyncIntervalInSeconds` (syncs every N seconds, minimum 10 seconds). Default is every 10 seconds if no configuration is specified.
  • MessageWeight: Modifies the impact of a request against the quota. A MessageWeight of 2 counts as two requests (default is 1). Useful for batch requests where each item should count against the quota.

2.3. ResetQuota Policy

The `ResetQuota` policy is used to decrease the number of counted requests in a quota counter, effectively giving back allowance without regard to the regular quota time period.

  • Configuration: Specifies the `Quota` policy name and `Identifier` value to update. Use default if no identifier is used in the Quota policy.
  • Allow Element: A required configuration that specifies the number of counted requests to be removed from the current count.

3. Caching

Caching stores data temporarily to reduce the need for repeated fetching from the original source. In Apigee, caching can persist state between calls and eliminate unnecessary backend service calls.

Benefits: Improve backend stability, improve call latency, reduce cost of metered services.

Cache Structure: Apigee uses a two-level cache:

  • L1 Cache (In-Memory): Located in each message processor. Checked first, very fast. Caches entries for a short period (currently 1 second). Least recently used entries are removed if the cache fills up.
  • L2 Cache (Runtime Data Store): Persisted in a database (Cassandra). Slower than L1 but faster than network calls. Entries are removed based on their Time-To-Live (TTL) or if the maximum number of entries is reached. When an L2 entry is read, it's copied to the L1 cache.
  • Entry Updates/Deletes: When an entry is updated or deleted by a message processor, it affects both its L1 cache and the shared L2 data store. Other message processors' L1 caches might hold stale data until their TTL expires.
  • Multi-Region: L2 changes are propagated via database replication. Consider stale L1 data and L2 replication delays in multi-region architectures.
  • Automatic L1 Caching: Apigee automatically caches OAuth access tokens, developers, developer apps, API products, and their custom attributes in L1 cache for 180 seconds.

3.1. General-Purpose Caching Policies

These policies are used to store and retrieve arbitrary strings.

PopulateCache Policy: Adds an entry to a cache.
  • CacheResource: Logical name of the cache (implicitly created, scoped at environment level).
  • CacheKey: Generated from a `Prefix` and `KeyFragment`'s. Double underscores are used as separators.
    • `Prefix`: Scopes the entry so it can only be retrieved using the same prefix.
    • `Scope`: Used if `Prefix` is not set, automatically generates a prefix based on the scope:
      • `Global`: Accessible by all proxies in the environment (`org_env`).
      • `Application`: Accessible by all endpoints in the current proxy (`org_env_proxy`).
      • `Exclusive`: Accessible only by the current proxy and endpoint (`org_env_proxy_endpoint`).
  • ExpirySettings: Sets the entry's expiration by date (`Expiry Date`), time of day (`TimeOfDay`), or number of seconds (`TimeoutInSeconds`).
  • Source: Name of the variable containing the string value to be stored.
LookupCache Policy: Looks for an entry in a cache.
  • Uses the same `CacheResource`, `CacheKey` (with `Prefix` or `Scope`) as `PopulateCache`.
  • AssignTo: Name of the output variable to populate with the retrieved value.
  • CacheLookup TimeoutInSeconds: Timeout for the cache lookup.

9 API Publishing and Developer Portals

This document summarizes key concepts regarding API publishing, API versioning, and developer portals, based on the provided material.

API Versioning

API versioning is a crucial aspect of API management, serving as a contract between the API provider and its consumers.

What API Versioning Is NOT:

  • Not an indicator of internal release version: API versions are distinct from internal software release cycles or API proxy revisions. Consumers are not concerned with internal development specifics.
  • Not something to do frequently: Changing an API's version has a significant impact on consumers, often requiring changes in their applications. Most API modifications can be made without updating the version.

What API Versioning IS:

  • A contract: Once an app is built against a specific API version, it should continue to function even if the API's functionality is enhanced or altered, as long as backward-compatibility is maintained.
  • A way to communicate breaking changes: A new API version should only be introduced when breaking changes occur. These include adding new required inputs to requests or removing previously available data from responses.
  • URL-based Versioning Scheme:
    • A common approach where the API version is specified in the URL (e.g., `hostname/api/version/resources`).
    • Version APIs separately: If you have multiple APIs, they should generally be versioned independently.
    • Avoid minor versions: Minor versions (e.g., 1.2) imply frequent or incremental changes, which contradicts the infrequent and impactful nature of API version changes.
    • APIs might always be version 1: An API can remain at version 1 even with changes, as long as those changes maintain backward-compatibility.
    • Consistency is key: While URL-based versioning is common, consistency across your APIs is most important, regardless of the chosen scheme.

Version Lifecycle and Deprecation

Managing multiple API versions and deprecating older ones requires a structured approach to minimize disruption for developers.

  • Limit live versions: It's generally recommended to have no more than two live versions of an API available simultaneously. Maintaining too many active versions is difficult and discourages upgrades.
  • Transition period: Aim for a transition period of no more than six months before fully deprecating an older version. This provides sufficient time for developers to make necessary adjustments.
  • Phases of upgrade and deprecation:
    1. Announce: Communicate the upcoming new version to the developer community. Provide details on changes, reasons for the changes, and availability. Highlight any benefits for developers and explain the upgrade process.
    2. Launch & Maintain: Launch the new API version and continue to promote it. Address any issues that arise during usage or upgrading promptly to ensure a smooth transition for developers.
    3. Deprecate: Clearly communicate the deprecation date for the previous version. Monitor analytics to identify apps still using the old version and reach out to developers to encourage migration before their apps stop working. Eventually, the old version can be completely turned off, with error messages directing users to the new version.

Developer Portals

Developer portals are essential for enabling app developers to discover, learn about, and integrate with your APIs.

  • App developers as customers: APIs should be designed with the app developer in mind, treating them as customers.
  • Purpose of developer portals:
    • Learn about service offerings: Developers can understand the APIs available and their service levels.
    • Learn how to use APIs: Portals provide comprehensive, live documentation, often created by linking OpenAPI specifications to published API products.
    • Register apps: Developers can self-service register their applications to use your API products.
    • Minimize time to first app: Rich documentation and self-signup processes accelerate a developer's ability to build an app using the API.
  • Types of developer portals:
    • Integrated portal: A lightweight, self-service portal, often hosted by API management platforms like Apigee.
    • Drupal 8 portal: A fully customizable self-service portal built using Drupal, an enterprise-level content management system. It integrates with API management platforms using specific modules, offering high flexibility but requiring more effort.
    • Do-it-yourself custom portal: Building a portal from scratch on any platform and integrating it using API management APIs. This option offers maximum flexibility but demands a very high level of effort and complexity.

Quiz Review (Key takeaway from the quiz)

The version of an API should be changed when there are breaking changes, such as adding a new required parameter to an existing API call. It is not tied to proxy revisions, adding new resources, or backend service version changes.

Beyond REST

gRPC

gRPC, which stands for gRPC Remote Procedure Call (where the "g" playfully changes with each version), is a modern, high-performance, open-source framework developed by Google. It's designed to enable efficient communication between services, particularly in distributed systems like microservices architectures.

What is gRPC?

Simple and Clear Explanation:

At its core, gRPC is an RPC (Remote Procedure Call) framework. This means it allows a program to call a function or method in another program located on a different machine (a "remote" machine) as if it were a local function call. gRPC handles all the complexities of network communication, data serialization, and deserialization behind the scenes.

Think of it like this: If you have a function `getUserProfile(userId)` in one service (say, a user service) and another service (say, an order service) needs to get user profile details, instead of making an HTTP request to `GET /users/{userId}`, the order service can directly "call" `getUserProfile(userId)` as if it were a local function. gRPC takes care of sending that "call" over the network to the user service and bringing the result back.

Key Features that Make gRPC Powerful:
  1. HTTP/2 as Transport: Unlike traditional REST APIs that often use HTTP/1.1, gRPC is built on HTTP/2. This brings significant benefits:
    • Multiplexing: Multiple requests and responses can be sent concurrently over a single TCP connection, reducing overhead and improving latency.
    • Header Compression: HPACK compression reduces the size of request and response headers.
    • Server Push: (Though less directly used in typical RPCs, it's an HTTP/2 feature).
  2. Protocol Buffers (Protobuf) for Data Serialization:
    • Interface Definition Language (IDL): You define your service methods and message structures in `.proto` files using Protobuf's IDL. This acts as a contract between client and server.
    • Binary Format: Protobuf serializes structured data into a compact, efficient binary format (not human-readable like JSON or XML). This makes messages smaller and faster to transmit and parse, contributing to gRPC's high performance.
    • Code Generation: From the proto file, gRPC tools automatically generate client "stubs" and server "skeletons" in various programming languages (Java, Python, Go, C++, Node.js, etc.). This means developers don't write boilerplate code for networking or serialization, saving time and reducing errors.
  3. Streaming Capabilities: Beyond simple request-response (unary calls), gRPC supports various streaming patterns:
    • Server-side streaming: Client sends one request, server sends back a stream of responses.
    • Client-side streaming: Client sends a stream of requests, server sends back one response.
    • Bidirectional streaming: Both client and server send independent streams of messages concurrently, enabling real-time communication (e.g., for chat applications or live data feeds).
Example:

Let's define a simple Greeter service in a proto file:

// greeter.proto
syntax = "proto3";

package helloworld;

// The greeting service definition.
service Greeter {
  // Sends a greeting
  rpc SayHello (HelloRequest) returns (HelloReply) {}
}

// The request message containing the user's name.
message HelloRequest {
  string name = 1;
}

// The response message containing the greetings.
message HelloReply {
  string message = 1;
}
How it works:
  1. Define (.proto): You define the Greeter service and Hello Request/Hello Reply messages in `greeter.proto`.
  2. Generate Code: You use the `protoc` compiler (with gRPC plugins) to generate client and server code in your chosen language (e.g., Python, Java).
  3. Implement Server: In your server-side code, you implement the `SayHello` method as defined in the Greeter service.
    4. # Simplified Python server implementation
class GreeterServicer(greeter_pb2_grpc.GreeterServicer):
    def SayHello(self, request, context):
        return greeter_pb2.HelloReply(message=f"Hello, {request.name}!")
  4. Implement Client: In your client-side code, you create a gRPC channel to the server and use the generated client stub to call the `SayHello` method as if it were a local function.
    5. # Simplified Python client implementation
with grpc.insecure_channel('localhost:50051') as channel:
    stub = greeter_pb2_grpc.GreeterStub(channel)
    response = stub.SayHello(greeter_pb2.HelloRequest(name='World'))
    print("Greeter client received: " + response.message)

When the client calls `stub.SayHello()`, gRPC handles:

  • Serializing HelloRequest to binary.
  • Sending it over HTTP/2 to the server.
  • The server deserializing it, executing its SayHello implementation.
  • The server serializing HelloReply to binary.
  • Sending it back to the client over HTTP/2.
  • The client deserializing it into a HelloReply object.

Apigee Support for gRPC

Apigee X (the cloud-native version on GCP) offers support for managing gRPC APIs, though with some specific considerations due to the nature of gRPC's binary communication.

Simple and Clear Explanation of Apigee's gRPC Support:

Apigee primarily provides passthrough support for gRPC. This means that while Apigee can act as a gateway for gRPC traffic, routing it from the client to your gRPC backend service, it generally does not inspect or modify the binary gRPC payload itself in the same way it can with text-based REST (JSON/XML) payloads.

How Apigee Manages gRPC:
  1. gRPC API Proxy: You create an Apigee API Proxy specifically configured for gRPC traffic. This proxy exposes a public endpoint that clients can call.
  2. TargetServer for gRPC Backend: You define a TargetServer in Apigee that points to your actual gRPC backend service. This TargetServer must have its protocol set to GRPC.
  3. Unary gRPC Requests: Currently, Apigee's direct gRPC proxy support primarily focuses on unary (request-response) gRPC calls. While HTTP/2 streams are used for the transport, Apigee treats the gRPC message as an opaque binary blob for direct proxying.
  4. No Payload Transformation Policies: Because Apigee doesn't parse the gRPC binary payload by default, policies that directly modify the request or response body (like XSLTTransform, JSONTOXML, ExtractVariables on content, AssignMessage to build a new body) cannot be used with gRPC proxies.
  5. Traffic Management & Security: You can still apply many Apigee policies based on metadata, headers, or other non-payload-affecting aspects:
    • API Key Verification: Use VerifyAPIKey policy.
    • OAuth2: Use OAuthV2 policy.
    • Quotas: Apply Quota policies (these will count total gRPC requests).
    • Spike Arrest: Use SpikeArrest policy.
    • CORS: Apply CORS policy.
  6. Contextual Information: Apigee exposes specific flow variables for gRPC, such as `request.grpc.rpc.name` (the name of the RPC method called) and `request.grpc.service.name` (the name of the gRPC service). You can use these variables in conditional flows or for analytics.
  7. Analytics: Apigee captures analytics for gRPC traffic, allowing you to monitor `x_apigee_grpc_rpc_name`, `x_apigee_grpc_service_name`, and `x_apigee_grpc_status`.
  8. ExternalCallout Policy (for advanced scenarios): For situations where you do need to inspect or modify gRPC payloads, or implement custom logic, Apigee offers the External Callout policy. This policy allows your API proxy to make a gRPC call to a custom gRPC server (which you implement and host) from within the proxy flow. Your custom gRPC server can then receive flow variables, perform operations (including parsing the gRPC payload if you pass it), and return modified variables or a response to Apigee. This gives you immense flexibility but requires external development.
Example: Securing a gRPC API with Apigee

Let's say you have a gRPC backend service at `my-grpc-backend.com:8080`.

  1. Define TargetServer: 
    <TargetServer name="my-grpc-backend-server">
  <Host>my-grpc-backend.com</Host>
  <Port>8080</Port>
  <IsEnabled>true</IsEnabled>
  <Protocol>GRPC</Protocol>
</TargetServer>
  2. Create gRPC API Proxy (simplified): 
    <ProxyEndpoint name="default">
  <HTTPProxyConnection>
    <BasePath>/my-grpc-service</BasePath>
    <VirtualHost>default</VirtualHost>
  </HTTPProxyConnection>
  <Flows>
    <PreFlow name="PreFlow" >
      <Request>
        <Step>
          <Name>VerifyAPIKey</Name>
        </Step>
        <Step>
          <Name>QuotaPolicy</Name>
        </Step>
      </Request>
      <Response/>
    </PreFlow>
  </Flows>
  <TargetEndpoint>
    <HTTPTargetConnection>
      <URL>grpc://my-grpc-backend-server</URL>
    </HTTPTargetConnection>
  </TargetEndpoint>
</ProxyEndpoint>

Now, when a gRPC client calls `https://your-apigee-domain.com/my-grpc-service` (the Apigee proxy endpoint), Apigee will:

  1. Receive the gRPC request (which is over HTTP/2).
  2. Execute the VerifyAPIKey policy to ensure a valid API key is present.
  3. Execute the QuotaPolicy to enforce rate limits.
  4. Proxy the gRPC request (the binary payload) directly to your `my-grpc-backend.com:8080` gRPC service.
  5. Receive the gRPC response from your backend and send it back to the client.

This demonstrates how Apigee provides a powerful and flexible layer to manage, secure, and monitor your gRPC services, even with its passthrough model for the binary content.

GraphQL

GraphQL is a powerful and increasingly popular alternative to traditional REST APIs, particularly for applications that need flexible data fetching and aggregation. Apigee, as a comprehensive API management platform, has strong support for managing GraphQL APIs.

Let's break down GraphQL itself and then how Apigee fits in.

What is GraphQL?

Simple and Clear Explanation:

GraphQL is a query language for your API and a runtime for fulfilling those queries with your existing data. Unlike REST, where you typically hit multiple endpoints to get different pieces of data (and often get more data than you need), GraphQL allows clients to:

  1. Request exactly what they need: Clients specify the exact data fields they want from a single endpoint, reducing "over-fetching" (getting too much data) and "under-fetching" (needing multiple requests to get all necessary data).
  2. Get predictable results: The structure of the GraphQL query often mirrors the structure of the JSON response, making it very intuitive.
  3. Define a schema: GraphQL APIs are built around a strongly typed schema that describes all possible data and operations. This schema acts as a contract between the client and server, enabling powerful introspection and tooling (like auto-completion in IDEs).
  4. Use a single endpoint: All GraphQL operations (queries, mutations, subscriptions) typically go to a single URL via an HTTP POST request.
Example:

Imagine you have a backend with users, products, and orders.

REST approach:

To get a user's name, their last 3 orders, and the names of the products in those orders, you might need:

  • `GET /users/{id}` (to get user name, but maybe also full address, email, etc.)
  • `GET /users/{id}/orders` (to get all orders)
  • For each order, `GET /orders/{orderId}/products` (to get product details)

This results in multiple HTTP requests and potential over-fetching of data.

GraphQL approach:

With GraphQL, you could make a single POST request to `your-graphql-api.com/graphql` with a query like this:

query GetUserData {
  user(id: "123") {
    name
    orders(last: 3) {
      id
      totalAmount
      products {
        name
      }
    }
  }
}

The response would be exactly what you asked for:

This demonstrates the efficiency and flexibility of GraphQL.

Apigee Support for GraphQL

Simple and Clear Explanation:

Apigee provides native support for managing GraphQL APIs, treating them as first-class citizens alongside REST APIs. This means you can apply Apigee's robust API management capabilities - such as security, traffic management, analytics, and developer monetization - to your GraphQL endpoints.

Apigee sits in front of your GraphQL backend server (which is responsible for "resolving" the data by fetching it from databases, other REST APIs, microservices, etc.). Apigee acts as the gateway, providing essential management features before the GraphQL query even reaches your resolver logic.

Key Apigee features for GraphQL:
  1. GraphQL Policy: This is a specialized policy in Apigee designed specifically for GraphQL requests. It allows you to:
    • Parse: Extract the GraphQL operation type (query, mutation, subscription), operation name, and fragment names into flow variables. This is crucial for applying conditional policies.
    • Verify: Validate the incoming GraphQL request payload against a pre-uploaded GraphQL schema. This ensures only schema-compliant requests reach your backend, saving backend CPU cycles and preventing malformed queries.
    • Impose restrictions: Control query complexity by setting limits on:
      • MaxDepth: The maximum nesting level of a query. This prevents overly deep and potentially expensive queries.
      • MaxCount: The maximum number of fragments (or distinct fields) in a query. This prevents overly broad queries that could return too much data.
    • Integrate with other policies: Once parsed, you can use standard Apigee policies based on the extracted GraphQL variables.
  2. Security (API Keys, OAuth2): You can secure your GraphQL API proxies using VerifyAPIKey and OAuthV2 policies, just like with REST. Apigee handles the authentication at the API gateway level before forwarding the request to your GraphQL backend.
  3. Traffic Management (Quotas, Spike Arrest): Apply quotas to GraphQL API Products and Spike Arrest policies to protect your GraphQL backend from traffic surges. These policies can be applied to the entire GraphQL API or even conditionally based on the parsed GraphQL operation type.
  4. API Products & Developer Portal: Bundle your GraphQL APIs into API Products, expose them through a Developer Portal, and manage developer access and subscriptions. The Developer Portal can even embed a GraphQL Explorer (GraphiQL), allowing developers to interactively build and test queries against your schema.
  5. Analytics: Gain insights into your GraphQL API usage, traffic patterns, and performance, just like with REST APIs. Apigee's analytics can show you which GraphQL operations are most popular, their latency, and error rates.
Example: Using the Apigee GraphQL Policy

Let's say you have a GraphQL API and want to prevent excessively deep queries, validate against your schema, and apply a quota.

    <Step><Name>VerifyAPIKey</Name></Step>

      <Name>GraphQLValidation</Name>
    </Step>
    <Step>
      <Name>Quota-for-Queries</Name>
      <Condition>graphql.operation.type = "query"</Condition>
    </Step>
    <Step>
      <Name>Quota-for-Mutations</Name>
      <Condition>graphql.operation.type = "mutation"</Condition>
    </Step>

<?xml version="1.0" encoding="UTF-8" standalone="yes"?>
<GraphQL name="GraphQLValidation">

  <SchemaFile>myGraphQLSchema.graphql</SchemaFile>

Explanation:
  1. When a client sends a GraphQL request to your Apigee proxy, the VerifyAPIKey policy checks for valid credentials.
  2. The GraphQLValidation policy then parses the GraphQL payload, populating flow variables like `graphql.operation.type` (e.g., "query", "mutation") and `graphql.operation.name`. It will also reject queries deeper than 8 levels or with more than 50 distinct fields.
  3. Finally, depending on whether it's a query or mutation, different quota policies can be applied conditionally using those flow variables.

This demonstrates how Apigee provides a powerful and flexible layer to manage, secure, and monitor your GraphQL APIs, offloading these concerns from your backend GraphQL server.
